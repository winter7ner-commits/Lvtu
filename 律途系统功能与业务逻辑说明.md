# 律途系统功能与业务逻辑说明

## 一、系统定位

律途系统是面向普通用户、律师和后台管理人员的在线法律服务平台。系统核心目标是让用户能够浏览法律法规、查询律师、发布法律服务订单，并由律师在线接单、提交服务结果，最终完成确认、评价与结算。后台管理端负责用户管理、法条维护、认证审核、订单监管、平台介入、评价治理和系统配置。

系统当前包含三个前端入口：

| 端口角色 | 使用对象 | 主要能力 |
| ---- | ---- | ---- |
| 游客 | 未登录用户 | 浏览首页、查看法条、查看律师信息；进入受限功能时提示登录注册 |
| 用户端 | 普通用户、律师账号 | 普通用户可发布服务、管理订单、评价律师；律师账号可进入接单大厅和我的接单 |
| 管理端 | 后台管理员 | 按后台角色进行审核、运营、客服介入、结算和系统配置 |

## 二、用户身份与权限规则

### 2.1 游客规则

游客是未登录状态下的访问身份。游客可以查看平台公开内容，但不能产生业务数据。

游客允许访问：

| 功能 | 说明 |
| ---- | ---- |
| 首页 | 查看平台首页、法律法规模块、常用服务入口 |
| 律师查询 | 浏览律师列表、律师详情、律师评价信息 |
| 法条浏览 | 查看法律分类、法律文件、具体法条、解释和评论 |
| 关于我们 | 查看平台介绍信息 |

游客受限功能：

| 功能 | 处理方式 |
| ---- | ---- |
| 发布服务 | 弹出登录提示，引导去登录或注册 |
| 我的订单 | 弹出登录提示 |
| 消息通知 | 弹出登录提示 |
| 收藏法条、发表评论、提交解释反馈 | 要求登录 |
| 申请成为律师 | 要求登录并完成实名认证 |

设计原因：游客可以先看到平台可提供的内容和服务价值，但真正涉及订单、认证、评价、消息等个人业务时必须登录。

### 2.2 普通用户规则

普通用户登录后可以使用用户端业务功能。普通用户的关键状态包括：

| 字段 | 说明 |
| ---- | ---- |
| `user_type = 1` | 普通用户 |
| `status = 1` | 账号正常 |
| `is_verified = true/false` | 是否已实名认证 |
| `auth_status` | 律师认证状态：0未申请，1审核中，2通过，3拒绝 |

普通用户可以：

| 功能 | 条件 |
| ---- | ---- |
| 修改个人信息 | 登录后可操作 |
| 法条收藏、评论、解释反馈 | 登录后可操作 |
| 实名认证申请 | 登录后可提交 |
| 发布服务订单 | 必须完成实名认证 |
| 指定律师发起咨询 | 必须完成实名认证 |
| 申请成为律师 | 必须完成实名认证 |
| 查看我的订单 | 只能查看自己发布的订单 |
| 确认服务结果、申请修改 | 只能操作自己的订单 |
| 发表评价 | 订单进入待评价后才能评价 |

普通用户不能：

| 限制 | 原因 |
| ---- | ---- |
| 进入律师端接单页面 | 不是律师账号 |
| 接单、提交服务结果 | 只有律师可操作 |
| 进入管理端 | 不是后台管理员 |

### 2.3 律师账号规则

律师账号本质上也是用户账号，但律师认证通过后具备接单能力。

律师账号关键条件：

| 条件 | 说明 |
| ---- | ---- |
| `user_type = 2` | 律师账号 |
| `auth_status = 2` | 律师认证通过 |
| `lawyer.status = 1` | 律师状态正常 |

律师端导航设计为：

```text
首页｜接单大厅｜我的接单｜消息通知｜关于我们
```

律师可以：

| 功能 | 说明 |
| ---- | ---- |
| 查看接单大厅 | 查看普通待接单订单 |
| 查看指定订单 | 用户指定该律师的订单只允许该律师接单 |
| 接单 | 订单必须处于待接单状态 |
| 查看我的接单 | 按状态筛选自己已接或被指定的订单 |
| 提交服务结果 | 订单进入处理中后可提交 |
| 重新提交服务结果 | 用户申请修改后，订单回到处理中，律师可重新提交 |
| 查看历史提交结果 | 律师可以查看自己每一次提交的服务结果快照 |
| 接收消息通知 | 包括被指定订单、用户申请修改、订单进入平台介入等 |

律师不能：

| 限制 | 原因 |
| ---- | ---- |
| 发布用户服务订单 | 避免律师账号既做服务方又做需求方 |
| 接非指定给自己的指定订单 | 指定订单只允许目标律师接单 |
| 修改非本人接单的结果 | 保护订单归属 |

## 三、注册、登录与账号状态

### 3.1 注册规则

用户注册时需要填写用户名、密码、手机号，邮箱可选。

| 字段 | 规则 |
| ---- | ---- |
| 用户名 | 必填；3到6个字符；只能包含字母、数字、下划线和连字符；不能有空格 |
| 密码 | 必填；6到16个字符；必须同时包含数字、小写字母和大写字母 |
| 手机号 | 必填；11位；符合 `1[3-9]` 开头；一个手机号只能注册一个账号 |
| 邮箱 | 非必填；填写时必须符合邮箱格式 |

默认测试密码统一为：

```text
Test123456
```

注册成功后：

| 字段 | 默认值 |
| ---- | ---- |
| `user_type` | 1，普通用户 |
| `status` | 1，正常 |
| `is_verified` | false，未实名认证 |
| `auth_status` | 0，未申请律师认证 |

### 3.2 登录规则

登录支持两种账号输入方式：

| 登录方式 | 示例 |
| ---- | ---- |
| 用户名 + 密码 | `user01 / Test123456` |
| 手机号 + 密码 | `13800000001 / Test123456` |

登录时会校验账号状态：

| 状态 | 说明 |
| ---- | ---- |
| 0 冻结 | 不允许登录，提示联系客服 |
| 1 正常 | 允许登录 |
| 2 封禁 | 不允许登录，提示联系客服 |
| 3 注销冷静期 | 当前代码允许登录，用户可取消注销 |
| 4 已注销 | 不允许登录 |

### 3.3 注销冷静期

用户可申请注销账号。申请注销后，账号进入冷静期。

业务规则：

| 规则 | 说明 |
| ---- | ---- |
| 冷静期天数 | 由后台系统设置配置，默认7天 |
| 冷静期内 | 用户可以取消注销 |
| 冷静期后 | 账号进入已注销状态，不再允许登录 |
| 不做找回功能 | 正式注销后不提供账号恢复入口 |

## 四、实名认证业务

实名认证用于约束发布订单、申请成为律师等高风险业务。

### 4.1 用户提交实名认证

用户需要填写：

| 字段 | 规则 |
| ---- | ---- |
| 真实姓名 | 2到20个字符 |
| 身份证号 | 18位，前17位数字，最后一位为数字或X |
| 手机号 | 必须与账号绑定手机号一致 |
| 身份证正面照 | 必须上传 |
| 身份证反面照 | 必须上传 |

提交后会生成实名认证记录，状态为：

```text
0 待审核
```

同时用户表中的 `is_verified` 保持为 `false`，直到后台审核通过。

### 4.2 后台审核实名认证

实名认证由后台的“超级管理员”或“认证审核员”审核。

审核结果：

| 操作 | 结果 |
| ---- | ---- |
| 通过 | 实名认证记录变为已通过，用户 `is_verified = true` |
| 驳回 | 实名认证记录变为被拒绝，用户 `is_verified = false`，必须填写驳回原因 |

### 4.3 测试演示规则

课程项目不接入公安实名认证接口，因此测试时使用格式正确的测试数据，由后台人工审核通过。

示例：

| 字段 | 示例 |
| ---- | ---- |
| 真实姓名 | 测试用户 |
| 身份证号 | 110101199001011234 |
| 手机号 | 当前账号绑定手机号 |
| 图片 | 任意测试图片 |

## 五、律师认证业务

普通用户完成实名认证后，才可以申请成为律师。

### 5.1 律师认证申请

申请人需要提交：

| 字段 | 说明 |
| ---- | ---- |
| 律师执业证号 | 必填，唯一 |
| 执业证照片 | 必填 |
| 所属律所 | 必填 |
| 执业开始年份 | 必填 |
| 申请类型 | 0首次入驻，1资料变更 |

申请状态：

| 状态 | 说明 |
| ---- | ---- |
| 0 待审核 | 用户已提交，等待后台审核 |
| 1 通过 | 律师认证通过，生成或更新律师信息 |
| 2 驳回 | 审核不通过，需要查看驳回原因 |

### 5.2 认证审核规则

律师认证由后台“超级管理员”或“认证审核员”处理。

审核通过后：

| 变化 | 说明 |
| ---- | ---- |
| 用户认证状态 | `auth_status = 2` |
| 用户身份 | 可作为律师使用律师端功能 |
| 律师信息 | 写入律师表，包括律所、执业年限、评分等 |

审核驳回后：

| 变化 | 说明 |
| ---- | ---- |
| 申请状态 | 变为驳回 |
| 用户认证状态 | 变为拒绝 |
| 驳回原因 | 后台填写，用户可查看 |

## 六、法律法规模块

法律法规模块提供法条浏览、解释、收藏、评论和解释反馈。

### 6.1 数据层级

法律法规采用三级结构：

```text
法律分类 -> 法律文件 -> 具体法条
```

示例：

```text
民法 -> 中华人民共和国民法典 -> 第一条
```

### 6.2 用户端法条详情

点击具体法条后，展示：

| 区域 | 内容 |
| ---- | ---- |
| 法条正文 | 条文编号、标题、正文 |
| 条文解释 | 对法条的解释说明 |
| 收藏 | 用户可收藏或取消收藏 |
| 评论 | 用户可查看评论、发表评论 |
| 解释反馈 | 用户可对解释选择“有帮助”或“无帮助”并提交原因 |

游客可以浏览法条详情，但收藏、评论、解释反馈需要登录。

### 6.3 解释反馈

用户在解释区域可以选择：

| 操作 | 说明 |
| ---- | ---- |
| 有帮助 | 可提交正向反馈，也可关闭弹窗不填 |
| 无帮助 | 可选择原因并补充说明，也可关闭弹窗不填 |

反馈进入后台“解释反馈”模块，由超级管理员或运营管理员处理。处理后状态从待处理变为已处理。

### 6.4 管理端法条管理

管理端可维护：

| 功能 | 说明 |
| ---- | ---- |
| 法律分类管理 | 新增、编辑、删除分类 |
| 法律文件管理 | 维护法规文件 |
| 法条管理 | 维护具体条文 |
| 法条浏览 | 管理员可查看用户端类似的法条详情 |
| 解释反馈处理 | 查看并处理用户提交的解释反馈 |

管理端查看法条详情时，不应作为普通用户发表评论或收藏。后台角色的职责是管理内容和处理反馈，不参与用户互动。

## 七、律师查询与评价展示

用户和游客都可以查询律师。

律师卡片展示：

| 信息 | 说明 |
| ---- | ---- |
| 律师姓名/账号 | 律师基础信息 |
| 所属律所 | 律师执业机构 |
| 擅长领域 | 如婚姻家事、合同审核、劳动争议 |
| 服务评分 | 根据评价综合得分展示 |
| 评价数量 | 展示该律师收到的评价数量 |
| 最近评价 | 展示用户选择律师时可参考的最近评价内容 |

业务目的：用户在选择律师前，不只看评分，也能看到评价数量和真实评价内容，类似生活服务平台的用户评价机制。

## 八、服务订单业务

### 8.1 发布订单前置条件

用户发布服务订单必须满足：

| 条件 | 说明 |
| ---- | ---- |
| 已登录 | 游客不能发布 |
| 普通用户账号 | 律师账号不能发布用户订单 |
| 账号正常 | 冻结、封禁、注销账号不能发布 |
| 已实名认证 | `is_verified = true` |

### 8.2 服务类型

系统支持多类法律服务，前端按服务类型显示不同表单。

常见类型包括：

| 类型 | 示例 |
| ---- | ---- |
| 在线咨询 | 用户填写咨询问题 |
| 电话咨询 | 用户填写联系方式和咨询时间 |
| 文书代写 | 用户填写文书需求 |
| 合同审核 | 用户上传或描述合同内容 |
| 诉讼服务 | 用户填写案件情况 |
| 婚姻家事 | 用户填写婚姻家庭相关需求 |

订单详情表保存不同服务类型对应的表单数据。

### 8.3 普通订单与指定律师订单

订单分为两种分配方式：

| 类型 | 字段 | 说明 |
| ---- | ---- | ---- |
| 普通订单 | `PUBLIC` | 支付后进入接单大厅，符合条件的律师可接单 |
| 指定律师订单 | `DIRECT` | 用户从律师详情发起咨询，支付后只允许目标律师接单 |

指定律师订单规则：

| 规则 | 说明 |
| ---- | ---- |
| 目标律师可见 | 指定律师会收到消息通知 |
| 接单限制 | 只有目标律师可以接单 |
| 其他律师 | 不允许接该订单 |

### 8.4 支付与订单状态

订单创建后初始状态为：

```text
待支付
```

支付成功后：

```text
待支付 -> 待接单
```

如果是指定律师订单，支付成功后会向目标律师发送消息通知。

### 8.5 订单完整状态流转

完整主流程：

```text
待支付
-> 待接单
-> 处理中
-> 待客户确认
-> 待评价
-> 已完成
```

各状态含义：

| 状态 | 说明 | 可操作角色 |
| ---- | ---- | ---- |
| 待支付 | 用户已创建订单但未支付 | 用户 |
| 待接单 | 已支付，等待律师接单 | 律师 |
| 处理中 | 律师已接单，正在处理；或用户申请修改后等待律师重新处理 | 律师 |
| 待客户确认 | 律师已提交服务结果，等待用户确认或申请修改 | 用户 |
| 待评价 | 用户确认服务完成，等待评价 | 用户 |
| 已完成 | 用户已评价，订单闭环 | 无 |
| 平台介入 | 修改次数达到上限或争议需要后台处理 | 客服专员/超级管理员 |

### 8.6 律师接单

律师接单必须满足：

| 条件 | 说明 |
| ---- | ---- |
| 律师存在 | 律师资料存在 |
| 律师状态正常 | `lawyer.status = 1` |
| 订单状态为待接单 | 其他状态不能接单 |
| 订单未被其他律师接单 | `lawyer_id` 为空 |
| 指定订单限制 | DIRECT订单必须由目标律师接单 |

接单成功后：

```text
待接单 -> 处理中
```

系统会给用户发送“订单已被律师接单”的消息通知。

## 九、服务结果提交与修改机制

### 9.1 首次提交服务结果

律师只能在订单状态为“处理中”时提交服务结果。

提交内容包括：

| 内容 | 说明 |
| ---- | ---- |
| 结果标题 | 必填 |
| 结果正文 | 服务处理说明 |
| 附件 | 可上传本地文件 |

提交成功后：

```text
处理中 -> 待客户确认
```

系统会：

| 处理 | 说明 |
| ---- | ---- |
| 保存当前服务结果 | 写入服务结果表 |
| 保存提交快照 | 记录本次提交的标题、内容、附件 |
| 通知用户 | 用户收到“律师已提交服务结果” |

提交快照用于保证用户和律师都能查看每一次提交的历史版本，而不是被后续修改覆盖。

### 9.2 用户确认结果

用户在“待客户确认”状态下可以选择：

| 操作 | 结果 |
| ---- | ---- |
| 确认完成 | 订单进入待评价 |
| 申请修改 | 订单回到处理中，律师重新处理 |

确认完成后：

```text
待客户确认 -> 待评价
```

### 9.3 用户申请修改

用户认为律师提交的服务结果有问题时，可以填写修改意见。

申请修改规则：

| 规则 | 说明 |
| ---- | ---- |
| 只能下单用户操作 | 其他用户不能申请 |
| 订单必须为待客户确认 | 未提交结果时不能申请修改 |
| 必须填写修改意见 | 修改意见不能为空 |
| 系统记录修改次数 | 每次申请生成一条修改记录 |
| 通知律师 | 律师收到修改通知 |

未超过次数上限时：

```text
待客户确认 -> 处理中
```

律师根据修改意见重新提交结果。

### 9.4 律师重新提交

律师收到修改申请后，订单处于“处理中”。律师可重新提交服务结果。

重新提交后：

```text
处理中 -> 待客户确认
```

系统会：

| 处理 | 说明 |
| ---- | ---- |
| 更新当前服务结果 | 当前结果变成最新版本 |
| 新增提交快照 | 保留本次重新提交内容 |
| 标记修改意见已处理 | 修改记录状态更新 |
| 通知用户 | 用户收到“律师已重新提交服务结果” |

前端展示规则：

| 内容 | 展示方式 |
| ---- | ---- |
| 首次提交结果 | 直接展示在提交结果区域 |
| 第一次修改后的提交 | 在修改记录中作为第1次重新提交，可查看详情 |
| 第二次修改后的提交 | 在修改记录中作为第2次重新提交，可查看详情 |
| 附件 | 每次提交对应自己的附件列表 |

### 9.5 修改次数限制与平台介入

修改次数由后台系统配置：

```text
order.max_revision_request_count
```

默认值：

```text
2
```

当用户申请修改次数达到上限后，再次申请修改会进入：

```text
平台介入
```

平台介入目的：

| 场景 | 说明 |
| ---- | ---- |
| 用户多次不满意 | 避免订单无限循环 |
| 律师认为已完成服务 | 双方对结果是否达标有争议 |
| 用户仍要求修改 | 超过约定修改次数 |
| 服务结果争议 | 需要平台根据记录协调 |

## 十、平台介入业务

平台介入由后台“客服专员”或“超级管理员”处理。

### 10.1 可查看内容

平台介入处理页应查看：

| 内容 | 说明 |
| ---- | ---- |
| 订单基本信息 | 用户、律师、订单金额、状态 |
| 最新服务结果 | 律师当前提交结果 |
| 历史提交记录 | 每一次提交和重新提交 |
| 用户修改意见 | 每一次申请修改内容 |
| 处理日志 | 平台处理动作和说明 |

### 10.2 处理动作

后台可执行：

| 动作 | 结果 |
| ---- | ---- |
| 要求律师继续处理 | 订单回到处理中 |
| 要求用户确认 | 订单回到待客户确认 |
| 标记待评价 | 订单进入待评价 |

处理后系统会记录平台处理日志，并向用户和律师发送消息通知。

## 十一、评价业务

### 11.1 发表评价

订单进入“待评价”后，用户可以发表评价。

评价包含三个评分维度：

| 评分项 | 范围 |
| ---- | ---- |
| 专业度评分 | 1到5星 |
| 响应速度评分 | 1到5星 |
| 服务态度评分 | 1到5星 |

其他字段：

| 字段 | 说明 |
| ---- | ---- |
| 综合评分 | 三个评分计算得出 |
| 评价内容 | 可选，最多500字 |
| 是否匿名 | 用户可选择匿名 |

业务限制：

| 规则 | 说明 |
| ---- | ---- |
| 一单只能评价一次 | 通过订单唯一约束限制 |
| 只能评价自己的订单 | 用户ID必须匹配 |
| 只能评价订单律师 | 律师ID必须匹配 |
| 订单状态必须允许 | 待评价或已完成 |

评价成功后：

```text
待评价 -> 已完成
```

### 11.2 评价展示

律师查询页面展示律师评价信息，包括：

| 内容 | 说明 |
| ---- | ---- |
| 综合评分 | 用户选择律师的重要参考 |
| 评价数量 | 反映服务样本数量 |
| 最近评价 | 展示最近用户反馈 |

### 11.3 评价管理

后台“超级管理员”或“运营管理员”可以管理评价。

可处理：

| 操作 | 说明 |
| ---- | ---- |
| 隐藏评价 | 对不适合展示的评价进行隐藏 |
| 删除评价 | 删除违规评价 |
| 恢复评价 | 恢复被隐藏或删除的评价 |
| 精选评价 | 标记优质评价 |
| 取消精选 | 取消精选标记 |
| 处理举报 | 对用户举报进行处理 |

所有后台评价操作都会进入管理员评价操作日志。

## 十二、消息通知业务

消息通知用于解决用户和律师必须手动查看订单状态的问题。

### 12.1 通知触发场景

## 十七、主要业务闭环总结

### 17.1 用户订单闭环

```text
用户实名认证
-> 发布服务订单
-> 支付
-> 律师接单
-> 律师提交服务结果
-> 用户确认或申请修改
-> 用户确认完成
-> 用户评价
-> 订单完成
```

### 17.2 律师入驻闭环

```text
普通用户注册
-> 完成实名认证
-> 提交律师认证申请
-> 后台认证审核员审核
-> 审核通过
-> 成为律师账号
-> 进入律师端接单
```

### 17.3 争议处理闭环

```text
律师提交结果
-> 用户申请修改
-> 律师重新提交
-> 用户再次申请修改
-> 达到修改次数上限
-> 进入平台介入
-> 客服专员处理
-> 订单回到处理中/待客户确认/待评价
```
